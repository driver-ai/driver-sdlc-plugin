# Open PR

Open a pull request from handoff documentation generated by `/drvr:docs-artifacts`.

### Step 1: Locate Feature

1. **If argument provided**: Use it as the feature path
2. **Else scan cwd**: Look for `FEATURE_LOG.md` in the current directory
3. **Scan parent directories**: Check up to 3 levels up for `FEATURE_LOG.md`
4. Read `FEATURE_LOG.md` to confirm feature exists and get the feature name

### Step 2: Check Prerequisites

Run these prerequisite checks in order. Each is a hard gate — stop on first failure.

1. **GitHub CLI authentication** — Run `gh auth status`:
   - If it fails: **BLOCK**: "GitHub CLI is not installed or not authenticated. Run `gh auth login` first."

2. **Handoff documentation** — Check if `driver-docs/` directory exists in the feature project:
   - If missing: **BLOCK**: "Handoff documentation has not been generated. Run `/drvr:docs-artifacts` first."

3. **Branch information** — Read Implementation Environment from `plans/00-overview.md` or per-plan `## Environment` sections to get:
   - Codebase path
   - Base branch
   - Feature branch

4. **Existing PR check** — Run `gh pr list --head <feature-branch> --state open --json url,number` from the codebase directory:
   - If a PR already exists: Report the existing PR URL and ask the user: "PR #N already exists for this branch. Update its body with current driver-docs, or skip PR creation?"
     - If user chooses **update**: Read driver-docs, compose body (Step 3-4), run `gh pr edit <number> --body <body>`, then skip to Step 6 (update FEATURE_LOG if `pr_created` event missing).
     - If user chooses **skip**: Exit with the existing PR URL.

### Step 3: Read Driver-Docs Artifacts

Read these files from `driver-docs/`:

1. `driver-docs/feature-overview.md`
2. `driver-docs/architecture.md`
3. `driver-docs/testing-guide.md`
4. `driver-docs/risk-assessment.md`

If any of the 4 expected files is missing, compose the PR body from what exists and note which sections are absent in the PR body. Log a WARN but do not BLOCK — a partial PR body is better than no PR.

### Step 4: Compose PR

1. **Title**: Synthesize from feature name — keep under 70 characters
2. **Body**: Compose with these sections:
   - `## Summary` — from feature-overview
   - `## Architecture` — condensed highlights from architecture doc
   - `## Test Plan` — from testing-guide
   - `## Risks` — from risk-assessment

### Step 5: Push and Create PR

**Run git and gh commands from the codebase directory** (read Path from Implementation Environment or plan `## Environment`).

1. Ensure branch is pushed: `git push -u origin <feature-branch>`
2. Create PR: `gh pr create --base <base-branch> --title <title> --body <body>`
3. Capture the PR URL from output
4. Do NOT update FEATURE_LOG or other artifacts until PR creation is confirmed successful

### Step 6: Update Artifacts

**Return to the feature project directory** for SDLC artifact updates.

1. Update `driver-docs/feature-overview.md` Related section: replace `[PR #{number}]({link})` placeholder with actual PR URL
2. Update `FEATURE_LOG.md`:
   - Append `pr_created` event row: `| <date> | PR opened — pr_created <URL> | — |`
   - Set phase to "PR Review"

### Step 7: Commit and Report

1. Commit SDLC artifact updates in the feature project directory: `"chore: PR opened — <feature name>"`
2. Report the PR URL to the user
